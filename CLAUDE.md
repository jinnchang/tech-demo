# Tech Demo 生成指南

当用户输入包含技术点 + 关键词（示例/demo/演示/原理/机制/怎么工作的/举个例子/跑通流程/验证方案）时，按本指南执行。

## 两种演示类型

| 维度         | 应用演示                                               | 原理演示                                                   |
| ------------ | ------------------------------------------------------ | ---------------------------------------------------------- |
| **定位**     | 「怎么用」——应用场景与流程                             | 「是什么」——内部机制与核心概念                             |
| **触发词**   | 示例、demo、演示、怎么用、举个例子、跑通流程、验证方案 | 原理、机制、概念解析、内部实现、怎么工作的、为什么这样设计 |
| **注释侧重** | 说明关键步骤做了什么                                   | 标注关键步骤对应的原理概念                                 |
| **目录命名** | `{tech}-app-demo`（kebab-case）                        | `{tech}-principle-demo`（kebab-case）                      |

## 工作流程

### 1. 理解技术点

分析用户指定的技术点，明确：这是什么、核心用途、使用场景。存在歧义时先与用户确认。

### 2. 场景选型

- **应用演示**：选择最能展示核心用法的应用场景。
- **原理演示**：先列出核心概念（技术本质、组成部件、关键机制、与相关技术区别），再选择演示场景。

场景要求：单文件不超过 200 行，聚焦技术点本身，避免额外业务逻辑。

### 3. 生成计划 + 选型展示

将代码生成拆解为清晰步骤，每步说明：目的、实现、必要性。存在多个方案时主动提供选项并推荐其一：

```markdown
### 待确认点 1：XXX 技术选型

方案 1：XXX
方案 2：XXX

基于 XXX 考虑，建议选择方案 1
```

### 4. 等待用户确认

展示：选定场景、步骤拆解、技术选型建议及待确认项。用户只确认部分项时，基于已确认项继续，未确认项使用推荐方案。用户拒绝时回到步骤 2 重新选型。

确认询问语：「以上方案是否符合你的需求？如有异议请指明；如无异议，将按推荐选型生成代码。」

### 5. 生成代码与 README

用户确认后，在当前工作目录下创建 demo 目录，生成完整可运行的示例代码及同级 README.md。若发现无法实现，暂停并说明障碍。

## 代码规范

### 语言选择

技术点有主导语言时使用该语言（如 React → TypeScript，asyncio → Python），无明确倾向时默认 Python。

### 依赖与文件

- **只引入必要库**。常用参考：HTTP 服务 → `flask`；网络请求 → `requests`；JSON → `json`（内置）
- 不修改其他已有文件
- README 与代码同级
- 纯算法/工具函数 → 单一文件；有 HTTP 交互 → `server.py` + `client.py`；多角色交互 → 加配置文件

### 代码质量

- 变量名体现语义（如 `fetch_user_data` 而非 `do_thing`）
- 异常简化处理，不影响流程可不处理
- 输入输出精简，只展示核心逻辑
- 所有代码必须可正常执行，无占位符或未完成的函数
- 注释：模块级说明用途，关键步骤说明目的（应用演示）或对应原理（原理演示），行内注释用于复杂逻辑
- 关键位置使用 `print` 打印调用流程和关键决策
- 避免过度封装，三个相似行优于提前抽成函数
- 原理演示代码为原理服务，不追求完整功能

## README.md 结构

README 与代码同级，按类型包含以下章节：

### 应用演示

目录命名：`{tech}-app-demo/`

```
{tech}-app-demo/
├── README.md
└── main.py   (或 server.py + client.py)
```

README 章节顺序：

| 章节          | 内容要求                     |
| ------------- | ---------------------------- |
| `## 场景示例` | 1-2 句话说明演示什么场景     |
| `## 环境准备` | 安装依赖的命令（代码块形式） |
| `## 文件结构` | 目录树（代码块形式）         |
| `## 运行测试` | 按顺序编号的步骤             |
| `## 预期输出` | 实际运行时的终端输出示例     |

### 原理演示

目录命名：`{tech}-principle-demo/`

```
{tech}-principle-demo/
├── README.md
└── main.py   (或 server.py + client.py)
```

README 章节顺序：

| 章节          | 内容要求                     |
| ------------- | ---------------------------- |
| `## 核心概念` | 1-2 句话说明技术本质         |
| `## 关键机制` | 1-2 句话说明核心流程原理     |
| `## 场景示例` | 1-2 句话说明演示什么原理     |
| `## 环境准备` | 安装依赖的命令（代码块形式） |
| `## 文件结构` | 目录树（代码块形式）         |
| `## 运行测试` | 按顺序编号的步骤             |
| `## 预期输出` | 实际运行时的终端输出示例     |

## 生成自检

代码生成完成后，逐项确认：

- [ ] 只生成必需文件，无多余文件
- [ ] 无占位符、TODO 或未实现的函数
- [ ] 核心流程可执行，无语法错误
- [ ] 注释标注了关键步骤的目的/原理
- [ ] 关键流程有 print 输出
- [ ] README 包含环境准备、运行步骤、预期输出
